Your First Insert

You write one line of code. Here is what happens in the next few milliseconds.

What is a database engine?

Think of your application like a car. You press the accelerator and it moves — but you have no idea what is happening under the hood with pistons, fuel injectors, and timing belts. A database engine is exactly that machinery under the hood for your data.

You call two lines of code. The engine does the rest.

C# var db = new Database("mydata.mde"); db.Insert("Users", row);
What this means

Line 1: Open (or create) a database file called mydata.mde on disk. Load its structure into memory.

Line 2: Store one row of data in the table called "Users". The engine will lock the table, serialize the row to bytes, insert it into the index, and optionally record it in the transaction log — all automatically.

💡
No server needed

Unlike MySQL or PostgreSQL, this engine is a library — it runs inside your app process. No installation, no port 3306, no config files. Just a .NET package and a file path.

Why does Table need a lock?

If two threads tried to insert at the same moment without a lock, they could corrupt the B+ tree. The ReaderWriterLockSlim allows many concurrent reads but serializes all writes.

Reading the actual Insert code

Here is the real Insert method from Table.cs. Every line does something important. The right column translates each piece into plain English.

Table.cs public void Insert(DataRow row, Transaction? transaction = null) { _lock.EnterWriteLock(); try { var key = GetPrimaryKey(row); var serialized = SerializeRow(row); if (transaction != null) transaction.LogInsert( _schema.TableName, key, serialized); _index.Insert(key, serialized); _nextRowId++; } finally { _lock.ExitWriteLock(); } }
Plain English

EnterWriteLock() — Grab an exclusive lock. No other thread can write (or read) until we are done.

GetPrimaryKey(row) — Extract the unique identifier for this row (e.g. the "Id" column value). This becomes the key in the index.

SerializeRow(row) — Convert the entire row object into a raw byte array so it can be stored on disk. This is serialization.

transaction.LogInsert() — If we are inside a transaction, record this operation in the WAL before touching the index. Safety first.

_index.Insert(key, serialized) — Actually place the key and value into the B+ tree.

finally ExitWriteLock() — Always release the lock — even if something threw an exception. The finally block guarantees this.

Every database does this

This pattern — validate, lock, serialize, index, log — is not unique to this mini-engine. MySQL, PostgreSQL, and SQLite all follow the same fundamental steps, just with more complexity layered on top.

🗄

MySQL / InnoDB

Uses a B+ tree (called a clustered index), row-level locking, and a redo log (equivalent to WAL). Same pattern, production-grade implementation.

🐘

PostgreSQL

Uses heap files + separate B-tree indexes, MVCC for concurrency, and a WAL for durability. Slightly different structure, identical goals.

🪶

SQLite

A single-file embedded database (just like this engine!). Uses a B-tree, a WAL journal, and a serialization format. The closest cousin to our mini-engine.

🌍
The big picture

Understanding this mini-engine means you now understand the skeleton of every relational database in existence. The algorithms — B+ trees, ACID transactions, write-ahead logging, and LRU caching — appear in every serious database engine.

The Index Tree

How the database finds any record in milliseconds, even among millions.

The problem with unsorted data

Imagine a filing cabinet with one million sheets of paper thrown in randomly. To find a specific sheet, you would have to check every single one — starting from the top — until you found it. That is exactly what a database does without an index: a full table scan.

😰

Without an index

Find user #500000 among 1 million users: check row 1, row 2, row 3 ... row 500000. Average 500,000 comparisons. Slow.

⚡

With a B+ tree index

Find user #500000 among 1 million users: follow the tree from root to leaf. About 20 comparisons. Fast — regardless of table size.

A B+ tree is like a library card catalog with shortcuts. The top-level cards point you to sections, the section cards point to aisles, and the bottom-level cards (the leaves) have the actual book locations — and they are all linked together left-to-right so you can browse in order.

📐
O(log n) explained simply

O(log n) means: every time you double the data, you only add ONE extra step. 1 million records takes about 20 steps. 1 billion records takes about 30 steps. This is why databases never slow down as they grow.

Two types of nodes: routers and data holders

A B+ tree has exactly two kinds of building blocks. Understanding them unlocks everything else.

Internal Node A internal node holds routing keys only. Like a sign in a library: “A–F go left, G–Z go right.” No actual data lives here.
Leaf Node A leaf node holds real key-value pairs — the serialized row data. Every leaf is connected to the next and previous leaf, forming a sorted chain perfect for range queries.

How a new key gets inserted

When you insert a new key, the tree finds the right leaf, drops the key in, and — if the leaf is too full — splits it in two and promotes the middle key upward. This is how the tree stays balanced automatically.

BPlusTree.cs — Insert public void Insert( object key, object? value) { lock (_lockObject) { var leaf = FindLeafNode(key); InsertIntoLeaf(leaf, key, value); if (leaf.KeyCount > _order - 1) SplitLeafNode(leaf); } }
Plain English

lock (_lockObject) — Grab a thread lock so no two inserts happen at exactly the same time and corrupt the tree structure.

FindLeafNode(key) — Walk down the tree from root to the correct leaf node by comparing the key with each internal node along the way. O(log n) comparisons.

InsertIntoLeaf(leaf, key, value) — Drop the new key and its serialized value into the correct sorted position within the leaf.

if (leaf.KeyCount > _order - 1) — Check if the leaf has too many keys. If _order is 4, then more than 3 keys means the node is overfull.

SplitLeafNode(leaf) — Split the overfull leaf into two halves. Promote the middle key to the parent internal node to keep the tree balanced.

The leaf node structure in code reveals the linked chain directly:

BPlusTreeNode.cs public class BPlusTreeLeafNode : BPlusTreeNode { public List<object?> Values { get; } public BPlusTreeLeafNode? Next { get; set; } public BPlusTreeLeafNode? Previous { get; set; } }
Plain English

Values — The actual data (serialized row bytes) stored at each key position in this leaf.

Next — A pointer to the next leaf node in sorted order. Enables forward scanning for range queries.

Previous — A pointer to the previous leaf. Together with Next, this forms a doubly-linked list of all leaves — so you can scan backwards too.

Why range queries are brilliant

The linked leaf chain is the secret weapon of B+ trees. For a range query like "give me all orders from January" — the tree finds the first matching key in O(log n) steps, then simply walks the leaf chain forward until it passes the end of the range. No revisiting internal nodes at all.

BPlusTree.cs — Range() public IEnumerable<KeyValuePair< object, object?>> Range(object? minKey, object? maxKey) { var leaf = minKey != null ? FindLeafNode(minKey) : GetFirstLeaf(); while (leaf != null) { for (int i = 0; i < leaf.Keys.Count; i++) { var key = leaf.Keys[i]; if (minKey != null && _comparer.Compare( key, minKey) < 0) continue; if (maxKey != null && _comparer.Compare( key, maxKey) > 0) yield break; yield return new KeyValuePair(key, leaf.Values[i]); } leaf = leaf.Next; } }
Plain English

FindLeafNode(minKey) — Jump straight to the leaf that would contain the minimum key. Skip all irrelevant leaves entirely using the tree structure.

while (leaf != null) — Walk forward through the linked leaf chain using the Next pointer. No need to go back up to the tree.

if key < minKey: continue — Skip any keys that landed in this leaf but are still below our range start.

if key > maxKey: yield break — As soon as we pass the end of the range, stop immediately. No more scanning needed.

yield return — Return each matching key-value pair to the caller one at a time as they are found — efficient streaming, no big list built in memory.

leaf = leaf.Next — Move to the next leaf in the chain. This is the linked list in action.

🔢
The math behind the magic

With _order = 4, each node holds up to 3 keys. The tree height is log base 3 of N. For 1 million records: log₃(1,000,000) ≈ 13 levels. For 1 billion: ≈ 19 levels. The page cache means most of those levels are already in memory, making lookups nearly instantaneous.

The single-responsibility principle in action

Each component does exactly one thing. Table does not know about disk. StorageEngine does not know about transactions. WALManager does not know about B+ trees. This separation makes each component testable, replaceable, and understandable on its own.

The codebase at a glance

The engine lives in the MiniDatabaseEngine project. Each file maps to exactly one responsibility. Here is the full cast, with what each file does.

Database.cs Main API entry point. new Database("file.mde") opens or creates a database. Holds all tables and coordinates every operation.
Table.cs One table = one instance. Owns the B+ tree index and a reader-writer lock. Handles Insert, Delete, Update, and Query for that table.
BPlusTree.cs The core index. Keeps keys sorted, provides O(log n) lookup, enables range scans via the leaf chain. The most algorithmically complex file in the project.
BPlusTreeNode.cs Defines the two node types: internal nodes (routing keys only) and leaf nodes (data + Next/Previous pointers for the linked chain).
StorageEngine.cs Reads and writes 4KB pages to the .mde file. Supports an optional memory-mapped mode for faster bulk access.
Page.cs A 4096-byte block — the basic unit of storage. Every node in the B+ tree and every row of data ultimately lives in one or more pages.
PageCache.cs An LRU cache of recently-used pages. Avoids re-reading the same page from disk repeatedly. A cache hit costs nanoseconds vs. microseconds for a disk read.
ExtentCache.cs A higher-level extent-level LRU cache. Caches 8 pages at a time, improving sequential read performance for large table scans.
Transaction.cs Represents one transaction. Tracks every operation performed within it so they can be committed together or rolled back entirely.
TransactionManager.cs Manages transaction lifecycle: begin, commit, rollback. Also handles crash recovery on startup by reading the WAL and undoing or replaying transactions.
WALManager.cs The Write-Ahead Log manager. Writes every operation to the WAL file before the change is applied to the B+ tree. This is the D in ACID: Durability.
WALEntry.cs One record in the WAL: type (Insert / Update / Delete / Commit / Rollback), the table name, key, old value, and new value. The transaction manager reads these on recovery.
DataSerializer.cs Converts C# objects to byte arrays ( serialization) and back. Handles 15 data types including int, string, DateTime, bool, decimal, and nullable variants.
TableQuery.cs A LINQ provider. Lets you write .Where(r => ...) style queries against a table, which the engine then translates into efficient B+ tree operations where possible.

Here is how all those files work together in the simplest possible usage — opening a database, running a transaction, and inserting a row safely.

Usage example (C#) var db = new Database("shop.mde"); var tx = db.BeginTransaction(); try { db.Insert("Products", row, tx); tx.Commit(); } catch { tx.Rollback(); throw; }
Which file handles each line

new Database("shop.mde") — Database.cs opens or creates the file, initializes the StorageEngine, PageCache, WALManager, and TransactionManager.

BeginTransaction() — TransactionManager.cs creates a new Transaction object and returns it. The WALManager is ready to log.

db.Insert("Products", row, tx) — Database.cs finds the right Table. Table locks, calls DataSerializer to serialize the row, calls WALManager to log the insert, then calls BPlusTree to index the key. StorageEngine and PageCache handle any necessary disk I/O.

tx.Commit() — TransactionManager.cs writes a COMMIT entry to the WAL via WALManager, making the change permanent. Even a crash now cannot lose this data.

tx.Rollback() — If anything went wrong, TransactionManager.cs reverses every operation logged for this transaction, restoring the database to its state before the transaction began.

Transactions: The Unbreakable Promise

How the database guarantees that a group of operations either all happen, or none of them do.

The Four Promises Every Database Makes

Imagine you are transferring $100 between bank accounts. Step one: debit Account A. Step two: credit Account B. Now imagine the server crashes after step one but before step two. The $100 has vanished. A transaction is the database's way of making this impossible.

Every transaction in a real database follows the ACID properties. These four words represent a contract between the database and every application that relies on it.

⚛️
Atomicity

All operations in a transaction succeed together, or none do. The transfer either fully completes or is fully undone — there is no in-between state.

✅
Consistency

Data always satisfies schema rules. Enforced via validation and serialization. A transaction cannot leave data in a broken state.

🔒
Isolation

Transactions do not interfere with each other. Implemented via a reader-writer lock — multiple readers allowed, but only one writer at a time.

🛡️
Durability

Once committed, data survives crashes. Guaranteed by writing to the WAL and flushing to disk before the commit returns.

💡
The Bank Transfer Test

Any time two operations must happen together or not at all, you need a transaction. Debits and credits, inventory decrements and order inserts, user creation and profile creation — these are all transaction-shaped problems.

The Database Keeps a Diary

Before doing anything, the database writes down what it is about to do. Think of a surgeon who writes the operation plan on a whiteboard before picking up a scalpel — if the power goes out mid-surgery, the next surgeon can read the whiteboard and know exactly what was in progress.

This diary is called the Write-Ahead Log. Every Insert, Update, and Delete is appended to a .wal file before being applied to the B+ tree. On commit, a Commit record is appended and the file is forcibly flushed to disk — only then does the commit return.

Transaction.cs public void Commit() { _lock.EnterWriteLock(); try { EnsureActive(); _walManager.AppendEntry(new WALEntry { TransactionId = _transactionId, OperationType = WALOperationType.Commit }); _walManager.Flush(); _state = TransactionState.Committed; } finally { _lock.ExitWriteLock(); } }
What This Does in Plain English
EnterWriteLock() — Grab exclusive access. Nobody else can write while we commit.
EnsureActive() — Sanity check: the transaction must not already be committed or rolled back.
AppendEntry(Commit) — Write a "Commit" record to the WAL diary on disk.
Flush() — Force the OS to actually write the diary to the physical disk. This is the moment durability is achieved.
_state = Committed — Only now is the transaction officially done. The Flush must succeed first.
ExitWriteLock() — Release exclusive access so other transactions can proceed.
📝
Why Flush() Is the Key Line

Without Flush(), the Commit record lives in the OS buffer — it might not reach disk before a crash. Flush() (or fsync) is the guarantee. It is the most expensive call in the commit path, which is why batching multiple writes into one transaction is much faster than committing each one individually.

Rollback: Reversing Time

When something goes wrong mid-transaction, the database needs to undo every change it already made — in reverse order. This is called a rollback. The last change made is the first one undone, like rewinding a tape.

The elegance is in the inversion: Insert becomes Delete, Delete becomes Insert, and Update simply swaps the old and new values. Every WAL entry stores both the new value AND the old value, making this possible.

Transaction.cs — Rollback() public void Rollback() { _lock.EnterWriteLock(); try { for (int i = _entries.Count - 1; i >= 0; i--) { var entry = _entries[i]; var undoEntry = CreateUndoEntry(entry); _undoCallback(undoEntry); } _walManager.AppendEntry(new WALEntry { TransactionId = _transactionId, OperationType = WALOperationType.Rollback }); _walManager.Flush(); _state = TransactionState.RolledBack; } finally { _lock.ExitWriteLock(); } }
What This Does in Plain English
Loop from last to first — Undo operations in reverse order. If we inserted A, then B, we undo B first, then A.
CreateUndoEntry(entry) — Produce the mirror-image operation: Insert → Delete, Update → Update with swapped values, Delete → Insert.
_undoCallback(undoEntry) — Actually apply the undo to the B+ tree, restoring the old data.
AppendEntry(Rollback) — Write a Rollback record to the WAL so recovery knows this transaction was intentionally undone.
Flush() + RolledBack — Same pattern as commit: force to disk, then update state.
Transaction.cs — CreateUndoEntry() private WALEntry CreateUndoEntry(WALEntry entry) { switch (entry.OperationType) { case WALOperationType.Insert: return new WALEntry { OperationType = WALOperationType.Delete, OldValue = entry.NewValue }; case WALOperationType.Update: return new WALEntry { OperationType = WALOperationType.Update, NewValue = entry.OldValue }; case WALOperationType.Delete: return new WALEntry { OperationType = WALOperationType.Insert, NewValue = entry.OldValue }; } throw new InvalidOperationException("Unknown op type"); }
The Elegant Inversion
Insert → Delete — We added a row, so undo it by deleting that same row (using its NewValue as the key).
Update → Update reversed — We changed a value, so undo it by changing it back (NewValue becomes OldValue).
Delete → Insert — We removed a row, so undo it by re-inserting the original row (stored in OldValue).
This works because WAL entries store BOTH old and new values — the old value is the safety net for undo.

Storage & Cache: Working Smarter

How the database reads as little as possible from disk — and why that matters.

Disk Is 1000x Slower Than Memory

Reading from a hard disk is like walking to a shelf in another building to fetch a book. Reading from memory is like reaching into your briefcase on the desk. The database spends enormous effort to avoid that trip across the building. Every trick in this module is about keeping data close.

The fundamental unit of storage is the page: a fixed 4096-byte (4KB) chunk. Every read and write touches a whole page, even if only a single byte changes. This mirrors how the operating system itself works.

Page.cs public class Page { public const int PageSize = 4096; public int PageId { get; set; } public byte[] Data { get; set; } public bool IsDirty { get; set; } }
What Each Field Means
PageSize = 4096 — Every page is exactly 4KB. This constant matches the OS memory page size, making reads and writes maximally efficient.
PageId — The page number, used to find it in the file. PageId 5 starts at byte offset 5 × 4096 = 20480 in the database file.
Data — The raw 4096 bytes. Records, index nodes, and metadata all live inside these bytes.
IsDirty — True when the in-memory page differs from what is on disk. Only dirty pages need to be written back, saving unnecessary I/O.
🐌
Disk Access

Millions of nanoseconds per read. Even SSDs are 10-100x slower than RAM. Spinning hard drives are 100,000x slower. Every disk read must count.

⚡
Memory (Cache) Access

Hundreds of nanoseconds per read. If a page is in the page cache, the database skips the disk entirely. This is the goal.

Extents: Buy a Whole Notepad

When you need one sheet of paper, you buy a whole notepad — because you will almost certainly need the neighboring sheets too. Databases do the same thing with extents. When one page is needed, the storage engine loads all 8 neighboring pages at once, because they are likely to be accessed soon.

This is called spatial locality: data stored near each other tends to be used near each other. A B+ tree range scan, for example, reads pages sequentially — loading extents amortizes the I/O cost across 8 pages instead of paying it 8 separate times.

Page 4KB — the smallest unit of I/O. One page = one disk read. Every record lives inside a page.
Extent 32KB — 8 pages loaded together. The cache stores extents, not individual pages, for efficiency.
Cache Configurable capacity. Measured in extents. A larger cache means more data stays in memory between queries.
Memory-Mapped Files An alternative: let the OS manage page loading automatically by treating the database file as if it were RAM.
💡
Why 4096 Bytes?

4KB matches the OS memory page size on most systems. When the database reads a 4KB page, the OS can handle it in one native memory operation with no padding or splitting. MySQL InnoDB uses 16KB pages, PostgreSQL uses 8KB — all are multiples of 4096 for the same reason.

LRU Cache: The Chef's Mise en Place

A great chef keeps the ingredients they use most often right next to the stove — within arm's reach. Rarely used ingredients stay in the pantry. When the counter is full and a new ingredient is needed, the one that has been sitting unused the longest is put back in the pantry to make room.

The LRU cache works exactly this way. When a page is accessed, it moves to the front of the list (it is now the most recently used). When the cache is full, the page at the back of the list (least recently used) is evicted. Pages that are accessed frequently naturally stay warm in memory.

PageCache.cs — Get() public Page? Get(int pageId) { if (_cache.TryGetValue(pageId, out var node)) { lock (_lockObject) { MoveToHead(node); } return node.Page; } return null; }
What This Does in Plain English
TryGetValue(pageId, out node) — Check if the page is already in the cache. This is a dictionary lookup — O(1), extremely fast.
lock (_lockObject) — The LRU list position must be updated thread-safely. The lock ensures two concurrent reads do not corrupt the list structure.
MoveToHead(node) — Mark this page as "most recently used" by moving it to the front of the doubly-linked list. This is the LRU maintenance step.
return node.Page — Cache hit: return the in-memory page. No disk access needed.
return null — Cache miss: the caller must fetch the page from disk and then add it to the cache.

Serialization: C# Objects Become Bytes

Everything stored in the database — strings, integers, dates — must be converted to raw bytes before it can be written to disk. This conversion is called serialization. The reverse — turning bytes back into typed objects — is deserialization.

The mini-database uses a simple but complete scheme: a leading byte signals whether the value is null, followed by type-specific bytes for the actual value. Every byte is accounted for precisely.

DataSerializer.cs — Serialize() public static byte[] Serialize(object? value, DataType dataType) { if (value == null) return new byte[] { 0 }; using var ms = new MemoryStream(); using var writer = new BinaryWriter(ms); writer.Write((byte)1); switch (dataType) { case DataType.Int: writer.Write((int)value); break; case DataType.String: writer.Write((string)value); break; case DataType.DateTime: writer.Write(((DateTime)value).ToBinary()); break; } return ms.ToArray(); }
What This Does in Plain English
if (value == null) return {0} — Null is encoded as a single byte with value 0. Short and unambiguous.
writer.Write((byte)1) — Non-null values start with byte value 1, the "present" marker. This is the null/non-null flag.
DataType.Int → Write((int)value) — Integers are written as exactly 4 bytes in little-endian order.
DataType.String → Write((string)value) — BinaryWriter encodes strings with a length prefix followed by UTF-8 bytes.
DataType.DateTime → ToBinary() — DateTime is converted to a 64-bit integer (ticks) so it can be stored as 8 fixed bytes.
ms.ToArray() — All bytes collected in MemoryStream are returned as a byte array for storage in the page.

🔑
The Null Marker Byte

Every value — even a non-null one — starts with a single byte: 0 for null, 1 for present. This one byte is what lets the deserializer handle nullable columns cleanly. Without it, there would be no way to distinguish a null string from an empty one.

Four Clever Storage Tricks

Every production database uses these same patterns. The names may differ, but the ideas are identical across MySQL, PostgreSQL, SQLite, and every other engine that needs to be fast and reliable.

📦
Extent Caching

Load 8 pages at once instead of one. Neighboring pages are almost always needed together — a B+ tree scan, for example, reads pages sequentially. Loading an extent amortizes the disk I/O cost across 8 pages.

🧠
LRU Eviction

When the cache is full, evict the page used least recently. Hot pages — those accessed again and again — stay in memory automatically. Cold pages get pushed out without any manual management.

🔤
Type-Coded Serialization

Each column value is encoded with a null marker byte, followed by type-specific bytes. The deserializer reads the marker first, then knows exactly how many bytes to consume. Compact, fast, and self-describing.

🗺️
Memory-Mapped Files

An alternative to manual page caching: map the database file directly into the process address space. The OS handles page loading and eviction automatically using its own virtual memory system.

⚡
The Real Cost of a Cache Miss

A cache miss means a disk read. On a spinning hard drive that is ~10ms. At 100 requests/second, just 10 cache misses per request = 10,000ms of I/O latency per second — more than the server can handle. Cache size is one of the most impactful database tuning parameters.

The Five Big Ideas

Every database engine — MySQL, PostgreSQL, SQLite, or this mini-database — is built from the same five foundational ideas. You now understand all of them. Here is how they fit together and what each one means for the apps you build.

🌳
B+ Tree Index

Metaphor: a book index with page numbers. Without it: scan every row. With it: jump straight to the answer. Impact: the difference between milliseconds and minutes at scale.

📝
Write-Ahead Log

Metaphor: the surgeon who writes the plan before cutting. Impact: crash safety for free — the database heals itself on restart by reading the diary.

⚛️
ACID Transactions

Metaphor: the bank transfer that either completes fully or not at all. Impact: money never disappears, orders never half-exist, carts never contain ghosts.

🧠
LRU Page Cache

Metaphor: the chef who keeps frequent ingredients within arm's reach. Impact: the difference between hitting RAM (nanoseconds) and hitting disk (milliseconds).

🔒
Reader-Writer Locks

Metaphor: a library — many people can read at once, but renovations close the floor. Impact: high read concurrency without sacrificing write correctness.

What You Can Do Differently Now

Knowing how the engine works changes how you design features. You no longer have to guess why something is slow or why data sometimes disappears. Here is the mental model in practice.

1
When to add an index

Any column you filter or sort on in a query with more than a few thousand rows needs a B+ tree index. Without it, every query reads every row. With it, the database jumps directly to matching rows. Rule of thumb: if a query runs slowly, an index is the first thing to check.

2
When to use a transaction

Any time two or more operations must succeed or fail together, wrap them in a transaction. Orders and inventory updates, user creation and profile creation, payment debits and credits — if one must not happen without the other, use a transaction.

3
How to debug missing data

If data disappears, the first question is: was Commit() called? Uncommitted transactions are rolled back on crash. The second question: did a rollback happen due to an exception that was silently swallowed?

4
How to think about performance

Slow reads: likely missing index or cache too small. Slow writes: likely too many small transactions (each one flushes the WAL). Slow under concurrency: check if a writer is holding the lock too long, blocking all readers.
