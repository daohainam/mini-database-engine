What is a database engine?

Think of your application like a car. You press the accelerator and it moves — but you have no idea what is happening under the hood with pistons, fuel injectors, and timing belts. A database engine is exactly that machinery under the hood for your data.

You call two lines of code. The engine does the rest.

C# var db = new Database("mydata.mde"); db.Insert("Users", row);
What this means

Line 1: Open (or create) a database file called mydata.mde on disk. Load its structure into memory.

Line 2: Store one row of data in the table called "Users". The engine will lock the table, serialize the row to bytes, insert it into the index, and optionally record it in the transaction log — all automatically.

💡
No server needed

Unlike MySQL or PostgreSQL, this engine is a library — it runs inside your app process. No installation, no port 3306, no config files. Just a .NET package and a file path.

Why does Table need a lock?

If two threads tried to insert at the same moment without a lock, they could corrupt the B+ tree. The ReaderWriterLockSlim allows many concurrent reads but serializes all writes.

Reading the actual Insert code

Here is the real Insert method from Table.cs. Every line does something important. The right column translates each piece into plain English.

Table.cs public void Insert(DataRow row, Transaction? transaction = null) { _lock.EnterWriteLock(); try { var key = GetPrimaryKey(row); var serialized = SerializeRow(row); if (transaction != null) transaction.LogInsert( _schema.TableName, key, serialized); _index.Insert(key, serialized); _nextRowId++; } finally { _lock.ExitWriteLock(); } }
Plain English

EnterWriteLock() — Grab an exclusive lock. No other thread can write (or read) until we are done.

GetPrimaryKey(row) — Extract the unique identifier for this row (e.g. the "Id" column value). This becomes the key in the index.

SerializeRow(row) — Convert the entire row object into a raw byte array so it can be stored on disk. This is serialization.

transaction.LogInsert() — If we are inside a transaction, record this operation in the WAL before touching the index. Safety first.

_index.Insert(key, serialized) — Actually place the key and value into the B+ tree.

finally ExitWriteLock() — Always release the lock — even if something threw an exception. The finally block guarantees this.

Every database does this

This pattern — validate, lock, serialize, index, log — is not unique to this mini-engine. MySQL, PostgreSQL, and SQLite all follow the same fundamental steps, just with more complexity layered on top.

🗄

MySQL / InnoDB

Uses a B+ tree (called a clustered index), row-level locking, and a redo log (equivalent to WAL). Same pattern, production-grade implementation.

🐘

PostgreSQL

Uses heap files + separate B-tree indexes, MVCC for concurrency, and a WAL for durability. Slightly different structure, identical goals.

🪶

SQLite

A single-file embedded database (just like this engine!). Uses a B-tree, a WAL journal, and a serialization format. The closest cousin to our mini-engine.

🌍
The big picture

Understanding this mini-engine means you now understand the skeleton of every relational database in existence. The algorithms — B+ trees, ACID transactions, write-ahead logging, and LRU caching — appear in every serious database engine.
