Transactions: The Unbreakable Promise

How the database guarantees that a group of operations either all happen, or none of them do.

The Four Promises Every Database Makes

Imagine you are transferring $100 between bank accounts. Step one: debit Account A. Step two: credit Account B. Now imagine the server crashes after step one but before step two. The $100 has vanished. A transaction is the database's way of making this impossible.

Every transaction in a real database follows the ACID properties. These four words represent a contract between the database and every application that relies on it.

⚛️
Atomicity

All operations in a transaction succeed together, or none do. The transfer either fully completes or is fully undone — there is no in-between state.

✅
Consistency

Data always satisfies schema rules. Enforced via validation and serialization. A transaction cannot leave data in a broken state.

🔒
Isolation

Transactions do not interfere with each other. Implemented via a reader-writer lock — multiple readers allowed, but only one writer at a time.

🛡️
Durability

Once committed, data survives crashes. Guaranteed by writing to the WAL and flushing to disk before the commit returns.

💡
The Bank Transfer Test

Any time two operations must happen together or not at all, you need a transaction. Debits and credits, inventory decrements and order inserts, user creation and profile creation — these are all transaction-shaped problems.

The Database Keeps a Diary

Before doing anything, the database writes down what it is about to do. Think of a surgeon who writes the operation plan on a whiteboard before picking up a scalpel — if the power goes out mid-surgery, the next surgeon can read the whiteboard and know exactly what was in progress.

This diary is called the Write-Ahead Log. Every Insert, Update, and Delete is appended to a .wal file before being applied to the B+ tree. On commit, a Commit record is appended and the file is forcibly flushed to disk — only then does the commit return.

Transaction.cs public void Commit() { _lock.EnterWriteLock(); try { EnsureActive(); _walManager.AppendEntry(new WALEntry { TransactionId = _transactionId, OperationType = WALOperationType.Commit }); _walManager.Flush(); _state = TransactionState.Committed; } finally { _lock.ExitWriteLock(); } }
What This Does in Plain English
EnterWriteLock() — Grab exclusive access. Nobody else can write while we commit.
EnsureActive() — Sanity check: the transaction must not already be committed or rolled back.
AppendEntry(Commit) — Write a "Commit" record to the WAL diary on disk.
Flush() — Force the OS to actually write the diary to the physical disk. This is the moment durability is achieved.
_state = Committed — Only now is the transaction officially done. The Flush must succeed first.
ExitWriteLock() — Release exclusive access so other transactions can proceed.
📝
Why Flush() Is the Key Line

Without Flush(), the Commit record lives in the OS buffer — it might not reach disk before a crash. Flush() (or fsync) is the guarantee. It is the most expensive call in the commit path, which is why batching multiple writes into one transaction is much faster than committing each one individually.

Rollback: Reversing Time

When something goes wrong mid-transaction, the database needs to undo every change it already made — in reverse order. This is called a rollback. The last change made is the first one undone, like rewinding a tape.

The elegance is in the inversion: Insert becomes Delete, Delete becomes Insert, and Update simply swaps the old and new values. Every WAL entry stores both the new value AND the old value, making this possible.

Transaction.cs — Rollback() public void Rollback() { _lock.EnterWriteLock(); try { for (int i = _entries.Count - 1; i >= 0; i--) { var entry = _entries[i]; var undoEntry = CreateUndoEntry(entry); _undoCallback(undoEntry); } _walManager.AppendEntry(new WALEntry { TransactionId = _transactionId, OperationType = WALOperationType.Rollback }); _walManager.Flush(); _state = TransactionState.RolledBack; } finally { _lock.ExitWriteLock(); } }
What This Does in Plain English
Loop from last to first — Undo operations in reverse order. If we inserted A, then B, we undo B first, then A.
CreateUndoEntry(entry) — Produce the mirror-image operation: Insert → Delete, Update → Update with swapped values, Delete → Insert.
_undoCallback(undoEntry) — Actually apply the undo to the B+ tree, restoring the old data.
AppendEntry(Rollback) — Write a Rollback record to the WAL so recovery knows this transaction was intentionally undone.
Flush() + RolledBack — Same pattern as commit: force to disk, then update state.
Transaction.cs — CreateUndoEntry() private WALEntry CreateUndoEntry(WALEntry entry) { switch (entry.OperationType) { case WALOperationType.Insert: return new WALEntry { OperationType = WALOperationType.Delete, OldValue = entry.NewValue }; case WALOperationType.Update: return new WALEntry { OperationType = WALOperationType.Update, NewValue = entry.OldValue }; case WALOperationType.Delete: return new WALEntry { OperationType = WALOperationType.Insert, NewValue = entry.OldValue }; } throw new InvalidOperationException("Unknown op type"); }
The Elegant Inversion
Insert → Delete — We added a row, so undo it by deleting that same row (using its NewValue as the key).
Update → Update reversed — We changed a value, so undo it by changing it back (NewValue becomes OldValue).
Delete → Insert — We removed a row, so undo it by re-inserting the original row (stored in OldValue).
This works because WAL entries store BOTH old and new values — the old value is the safety net for undo.

Crash Recovery: Reading the Diary

When the database restarts after a crash, it reads the WAL from the beginning. For every transaction it finds, it falls into one of three categories. The recovery engine handles each one differently — this is how the database heals itself automatically.

Three Outcomes, One Algorithm

Committed = replay forward. Rolled back = skip. Uncommitted (crash) = undo in reverse. The WAL contains enough information to handle all three cases perfectly, which is why it must record both old and new values for every operation.
